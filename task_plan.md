# FLOYD Ecosystem Unification — Official Task Plan

## Goal

Determine, from direct inspection of every named FLOYD component and the live `ff` launch chain, the authoritative parts, boundaries, runtime topology, and process flows needed to make FLOYD one persistent workstation organism for coding, generative AI, multimedia, MCP, browser/mobile access, and multi-agent work; preserve the findings and an implementation-ready unification roadmap in this workspace.

## Success Criteria

- Every user-named component has an evidence-backed disposition: **adopt**, **extract**, **reference**, **quarantine**, or **retire**.
- The live `which ff` chain is traced to its actual executable, configuration, services, and state stores.
- The live `superfloyd` chain is traced and preserved as a second golden CLI oracle; its ASCII art is changed only in copied/new presentation code.
- One canonical control plane, runtime ownership model, protocol envelope, identity/session model, persistence model, and process lifecycle are defined.
- Desktop, CLI, Chrome bridge/extension, mobile PWA/tunnel, multimedia, sandbox, and multi-agent work share one explicit end-to-end flow.
- The roadmap is sequenced into buildable slices with acceptance gates, migration safety, and rollback boundaries.
- All conclusions distinguish repository evidence from recommendations and unresolved verification work.
- Former FLOYD features represented by the named harness, skill-system, brand, lab, and tunnel skills are redesigned as current platform capabilities rather than copied unchanged.
- The agent harness has a narrow typed action space, deterministic observation envelopes, explicit recovery contracts, and measurable completion quality.
- Skills are versioned, discoverable, load-on-demand capability packages with tests and cross-harness adapters, not prompt bulk embedded in every runtime.
- Brand voice remains a presentation policy that preserves factual content and explicit truth-state labels; it never changes runtime semantics or turns planned work into shipped work.
- Risky/untrusted execution is routed through an on-demand isolated lab provider with explicit mounts, secret boundaries, persistent/ephemeral modes, idle shutdown, and auditable lifecycle events.
- Connectivity recovery is a deterministic, single-flight supervisor: private paths first, bounded probes and safe actions, public fallback only behind explicit authorization, and one operator-facing status/next step.
- Mobile, SSH, coding, Browork dispatch, photo/video generation, and Git-level operations are first-class flows over the same runtime—not separate apps with separate brains.
- Every named source directory remains immutable. Any implementation work uses independently writable copies with provenance and copy verification; no hardlinks or writable symlinks may connect edits back to originals.
- Existing Floyd-on-OpenCode and branded editor assets are adjudicated before any new cockpit/editor build is planned.
- Delivery is private/team-only: no public product launch, public session sharing, or public raw tool/MCP exposure is part of this plan.
- Model routing reflects the actual machine and subscriptions: only tiny local-model roles are assumed; GLM and MiniMax annual plans are preferred, subscription and metered-API billing are mechanically distinguished, and silent paid fallback is prohibited.

## Source Protection and Copy Protocol

1. **Audit in place, read-only.** Inspection never requires a copy and never writes to a source.
2. **Select donors before copying.** Do not duplicate gigabytes of abandoned builds or dependencies until the disposition matrix identifies the exact donor trees.
3. **Check capacity first.** Record destination free space and source apparent/allocated size.
4. **Fingerprint the source.** Record absolute path, filesystem/device, Git HEAD/branch/status when present, size, and a manifest/hash for selected donor files.
5. **Create an independent copy.** Copy into this workspace's future `intake/<source-id>/` area with macOS metadata as needed. Never use hardlink cloning and never use a symlink as the editable tree.
6. **Verify separation.** Confirm representative source/copy files have different inode identities, symlinks do not escape to source, Git status matches, and checksum manifests agree before editing.
7. **Edit copies only.** Imported code moves from verified intake copies into canonical modules; originals stay untouched for rollback/provenance.
8. **Preserve dirty truth.** A Git clone alone is insufficient for dirty/untracked sources; copying must preserve the actual working tree plus `.git` metadata or separately archive the dirty delta.
9. **Use remotes as provenance, not replacement truth.** Compare remote history/tags with local working trees, but never discard local-only changes or lift a user-quarantined source merely because the remote is clean.
